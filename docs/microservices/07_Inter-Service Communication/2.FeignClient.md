# OpenFeign in Spring Boot

## 1. What OpenFeign is

OpenFeign is a declarative HTTP client framework for Java. 

In Spring Cloud, it is integrated so you can define a remote HTTP client as an interface and use Spring MVC-style annotations on its methods. 

Spring Cloud OpenFeign is currently treated as feature-complete, which means the project is in maintenance mode for bug fixes and small community changes rather than major new feature development.

---

## 2. Why teams use it

In real Spring microservices, Feign is popular because it keeps service-to-service calls readable and compact. 

Spring Cloud also wires Feign into Spring MVC annotations, Spring’s normal `HttpMessageConverters`, Eureka, Spring Cloud CircuitBreaker, and Spring Cloud LoadBalancer, so it fits naturally into the Spring ecosystem.

---

## 3. How to add it

Use the starter dependency `spring-cloud-starter-openfeign`. 

Spring Boot/Spring Cloud expects you to enable scanning with `@EnableFeignClients`, and you can point it at base packages or specific client interfaces.

Spring also supports using a direct `url` attribute on `@FeignClient` when you do not want service discovery.

```java
@SpringBootApplication
@EnableFeignClients(basePackages = "com.example.clients")
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

---

## 4. Mental model

A Feign client is not a controller and not a service implementation. It is a proxy that Spring creates for your interface, and that proxy calls a remote HTTP service for you. 

Spring Cloud creates a named client ensemble per `@FeignClient`, and that ensemble includes components such as a decoder, encoder, contract, and builder.

```text
Your interface
    ↓
Feign proxy
    ↓
HTTP client
    ↓
Remote service
    ↓
Decoder converts response to Java object
```

---

## 5. Default Spring Cloud OpenFeign building blocks

Spring Cloud OpenFeign provides default beans such as `ResponseEntityDecoder` wrapped around a Spring decoder, `SpringEncoder`, `Slf4jLogger`, `SpringMvcContract`, and a `FeignCircuitBreaker.Builder`. 

If Spring Cloud LoadBalancer is present, it uses `FeignBlockingLoadBalancerClient`; otherwise it falls back to the default Feign client. 

Spring Cloud OpenFeign also looks up optional beans like `Retryer`, `ErrorDecoder`, `Request.Options`, `RequestInterceptor`, `SetterFactory`, and `QueryMapEncoder` from the application context.

## 6. The most common `@FeignClient` pattern

```java
@FeignClient(name = "customer-service")
public interface CustomerClient {

    @GetMapping("/customers/{id}")
    CustomerResponse getCustomer(@PathVariable("id") Long id);

    @PostMapping("/customers")
    CustomerResponse createCustomer(@RequestBody CreateCustomerRequest request);

    @PutMapping("/customers/{id}")
    CustomerResponse updateCustomer(
            @PathVariable("id") Long id,
            @RequestBody UpdateCustomerRequest request);

    @PatchMapping("/customers/{id}")
    CustomerResponse patchCustomer(
            @PathVariable("id") Long id,
            @RequestBody PatchCustomerRequest request);

    @DeleteMapping("/customers/{id}")
    void deleteCustomer(@PathVariable("id") Long id);
}
```

Spring Cloud OpenFeign supports Spring MVC annotations and uses the same message converters Spring Web uses, so the interface feels very similar to a controller contract.

A banking service may need customer status before processing a transaction.

CustomerResponse customer = customerClient.getCustomer(customerId);

## 8. Headers, auth, and correlation IDs

Feign clients commonly need to propagate headers such as JWT tokens, tenant IDs, and correlation IDs. Spring Cloud OpenFeign supports `RequestInterceptor` beans, and Spring Cloud OpenFeign explicitly looks up a collection of request interceptors from the application context.

```java
@Configuration
public class FeignHeaderConfig {

    @Bean
    public RequestInterceptor authInterceptor() {
        return template -> {
            template.header("Authorization", "Bearer " + getToken());
            template.header("X-Correlation-ID", getCorrelationId());
            template.header("X-Tenant-ID", getTenantId());
        };
    }
}
```

Use this for:

* JWT propagation
* tenant propagation
* distributed tracing
* request auditing

## 9. Per-client configuration

Spring Cloud OpenFeign lets you customize each client with a separate configuration class. Spring also warns that if you annotate that config class with `@Configuration`, you should be careful not to let component scanning accidentally make it the default configuration for all Feign clients.

```java
@FeignClient(
        name = "customer-service",
        configuration = CustomerClientConfig.class
)
public interface CustomerClient {
    @GetMapping("/customers/{id}")
    CustomerResponse getCustomer(@PathVariable("id") Long id);
}
```

```java
public class CustomerClientConfig {

    @Bean
    public Logger.Level feignLoggerLevel() {
        return Logger.Level.FULL;
    }
}
```

This pattern is useful when one client needs:

* different logging
* different timeout values
* a special error decoder
* a custom retry policy

## 10. Error handling

OpenFeign and Spring Cloud OpenFeign support custom error handling. Spring Cloud OpenFeign looks up `ErrorDecoder` from the application context, and the underlying OpenFeign core uses `ErrorDecoder` plus retry behavior for certain retryable failures. Spring Cloud OpenFeign also creates a default `Retryer.NEVER_RETRY`, so Spring Cloud’s default behavior is not to retry unless you configure it. That is different from vanilla Feign, which retries I/O exceptions and retryable exceptions by default.

```java
@Configuration
public class CustomerFeignErrorConfig {

    @Bean
    public ErrorDecoder errorDecoder() {
        return (methodKey, response) -> {
            if (response.status() == 404) {
                return new CustomerNotFoundException("Customer not found");
            }
            if (response.status() == 409) {
                return new BusinessConflictException("Business conflict");
            }
            return new RuntimeException("Remote call failed: " + response.status());
        };
    }
}
```

### Common mapping in banking

* `404` → customer/account not found
* `409` → duplicate beneficiary, limit conflict, or business conflict
* `422` → business validation issue
* `503` → downstream service unavailable
* `500` → unexpected downstream failure

## 11. Timeouts and retries

Spring Cloud OpenFeign exposes `Request.Options` and related HTTP client settings as configurable pieces, and it supports Apache HttpClient 5 and OkHttp based clients. Spring Cloud OpenFeign also notes that Apache HttpClient 4 is no longer supported, and that HC5 / OkHttp are the modern options.

```java
@Configuration
public class CustomerTimeoutConfig {

    @Bean
    public Request.Options requestOptions() {
        return new Request.Options(
                2000,
                5000
        );
    }
}
```

Important production rule:

* keep connect timeout low
* keep read timeout bounded
* do not rely on defaults
* treat retries carefully so you do not duplicate non-idempotent operations

## 12. Load balancing and service discovery

If Spring Cloud LoadBalancer is on the classpath, Spring Cloud OpenFeign uses `FeignBlockingLoadBalancerClient`. With Eureka, the logical service name can resolve to real instances. If you do not use Eureka, Spring Cloud also supports simple static server lists through `SimpleDiscoveryClient`. Spring Cloud says LoadBalancer support is optional, so you must add the dependency when you need it.

```java
@FeignClient(name = "account-service")
public interface AccountClient {
    @GetMapping("/accounts/{id}")
    AccountResponse getAccount(@PathVariable("id") Long id);
}
```

In this setup:

* `account-service` is the logical service name
* discovery or load balancing resolves the actual instance
* the caller does not need to know the host and port

## 13. Logging

Spring Cloud OpenFeign provides `Slf4jLogger` by default and also lets you customize `Logger.Level`. This makes client-side request logging straightforward for debugging while still allowing you to keep production logs under control.

Typical log levels:

* `NONE`
* `BASIC`
* `HEADERS`
* `FULL`

Use `FULL` only in troubleshooting or lower environments because it can expose sensitive values.

## 14. Generic and paginated responses

Spring Cloud OpenFeign supports Spring MVC-style signatures, including collection returns and `Page` examples in the docs.

```java
@FeignClient(name = "statement-service")
public interface StatementClient {

    @GetMapping("/statements")
    Page<StatementResponse> getStatements(Pageable pageable);
}
```

Use this for:

* transaction history pages
* account statements
* audit trail pages
* report listings

Edge case:
if the remote service is not actually returning a Spring `Page` JSON shape, do not return `Page<T>` in the client. Match the response contract exactly.

## 15. Edge cases that matter in production

### 15.1 Lazy attribute resolution in tests

Spring Cloud OpenFeign 4.x resolves `@FeignClient` attributes eagerly by default, which is good for most use cases and AOT support. If you need lazy resolution, Spring Cloud provides the property `spring.cloud.openfeign.lazy-attributes-resolution=true`, and the docs specifically mention using lazy resolution for Spring Cloud Contract test integration.

### 15.2 Multi-module projects

In multi-module setups, Spring Cloud says you should specify the packages directly or list the clients explicitly in `@EnableFeignClients`, otherwise discovery can become messy.

### 15.3 Configuration classes and component scan

If your Feign configuration class is itself annotated with `@Configuration`, keep it out of broad component scanning unless you intentionally want it to become the default Feign config source.

### 15.4 Retries on money-moving APIs

Because Spring Cloud defaults to `Retryer.NEVER_RETRY`, retry is off unless you change it. That is often a good thing for non-idempotent operations like money transfer, where an automatic retry could create duplicate side effects.

```text
Transfer Service
    ↓
Customer Service
    ↓
Account Service
    ↓
Limit Service
    ↓
Fraud Service
    ↓
Ledger Service
```

A typical Feign-based implementation might look like:

```java
@FeignClient(name = "fraud-service")
public interface FraudClient {

    @PostMapping("/fraud/check")
    FraudCheckResponse check(@RequestBody FraudCheckRequest request);
}
```

```java
@FeignClient(name = "limit-service")
public interface LimitClient {

    @PostMapping("/limits/verify")
    LimitCheckResponse verify(@RequestBody LimitCheckRequest request);
}
```

Production note:
if `fraud-service` or `limit-service` is down, you usually do not want to keep retrying a transfer request blindly. Map the failure into a controlled business error or a safe fallback response.

## 17. Best industry usage pattern

The cleanest production pattern is:

```text
Controller
   ↓
Service
   ↓
Feign Client Interface
   ↓
ErrorDecoder / Interceptor / Timeout config
   ↓
Remote service
```

Keep Feign clients small and focused:

* one client per downstream domain
* DTOs in and out
* no business logic in the client interface
* centralize auth and tracing in interceptors
* centralize error mapping in `ErrorDecoder`

---

## 18. Common mistakes

Do not:

* put business logic inside the Feign interface
* forget timeouts
* retry non-idempotent calls without care
* log sensitive headers in `FULL` mode in production
* expose raw downstream exceptions directly to the caller
* use the wrong response shape for collections or pages
* let Feign config classes get pulled into global component scan accidentally

---

## 19. When to use Feign versus other clients

Feign is strongest when:

* you want declarative, readable clients
* you are in a Spring Cloud microservices stack
* you want built-in load balancing and Spring-style contracts
* you prefer interface-based client definitions over manual request code

---

## 20. Final summary

OpenFeign is the most common Spring-friendly synchronous service-to-service client style in many enterprise microservice systems. Spring Cloud OpenFeign is feature-complete, integrates with Spring MVC annotations, Spring Web message conversion, LoadBalancer, Eureka, and CircuitBreaker, and provides a strong extension model through `ErrorDecoder`, `RequestInterceptor`, `Logger.Level`, and `Request.Options`. Use it for clear, declarative HTTP clients, but handle timeouts, errors, retries, and sensitive headers carefully.
